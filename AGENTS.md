# 🤖 DailyFit AGENTS.md

## 1. 개발 환경 및 원칙

- **Style:** 함수형 컴포넌트 + React Hooks (Class 금지)
- **Library:** - 이미지 로딩은 무조건 `expo-image` 사용
  - 서버 상태는 `TanStack Query` 최우선
- **Comments:** 모든 주석은 한국어로, 함수/컴포넌트 단위로 명확하게 작성

## 2. 에이전트 작업 플로우 (필수 준수)

코딩 에이전트는 반드시 아래의 **[태스크 확인 및 브랜치 생성 ➔ 구현 ➔ 자체 리뷰 ➔ 사용자 테스트 ➔ Git 병합 및 다음 태스크]** 사이클을 엄격하게 반복하여 작업을 진행해야 합니다.

### Step 1: 태스크 확인 및 브랜치 생성 (Planning)

1. `docs/TASKS.md` 상단에서 진행해야 할 `[todo]` 작업을 하나 선택하여 `[doing]`으로 변경합니다.
2. 메인 브랜치(`main`)에서 최신 상태를 확인한 후, 해당 태스크 번호로 **새 기능 브랜치를 생성**합니다.
   - 브랜치 네이밍 규칙: `feature/T-XXX-short-description` (예: `feature/T-001-auth`)
   - 명령어: `git checkout -b feature/T-XXX-...`
3. `docs/PRD.md` 및 관련된 문서를 읽고 해당 태스크의 요구사항을 파악한 후, 구현할 범위와 방식을 사용자에게 공유하고 **승인을 받은 뒤 코딩을 시작**합니다.

### Step 2: 구현 (Coding)

1. **Zero-Inference:** 추측해서 기능을 추가하지 말고 합의된 범위의 내용만 구현합니다.
2. 미니멀 디자인 룰(Toss/Musinsa 스타일 반영, 여백 강조, 불필요한 장식 배제)을 철저히 준수합니다.
3. 모든 비동기 작업에는 Loading, Empty, Error 상태를 구현합니다.

### Step 3: 자체 리뷰 및 검증 (Review)

1. 코드를 작성한 후, 구현된 코드가 PRD 스펙과 정확히 일치하는지 스스로 점검합니다.
2. 불필요한 파일 수정, 공백 변경, 린트 에러가 없는지 확인합니다.
3. 빌드 및 렌더링에 문제가 없는지(터미널 로그 등) 확인 가능한 범위에서 점검합니다.

### Step 4: 사용자 테스트 요청 (User Testing)

작업이 일단락되면 사용자에게 테스트 및 리뷰를 요청해야 합니다. 이때 다음 양식으로 요약 보고합니다.

---

**[ 📝 작업 완료 보고 및 테스트 요청 ]**

- **수행 작업:** (태스크 번호 및 제목)
- **현재 브랜치:** (작업 중인 feature 브랜치명)
- **변경/생성 파일:** (수정된 파일 경로)
- **구현 핵심 요약:** (사용한 기술적 해결책, UI 컴포넌트 등)
- **사용자 테스트 가이드:** (사용자가 직점 테스트해야 하는 화면, 클릭할 버튼 등)

---

### Step 5: Git 병합 및 다음 태스크 (Next Task)

1. 사용자가 테스트 후 수정 사항(피드백)을 주면, 이를 즉각 반영하여 다시 Step 2~4를 진행합니다.
2. 사용자가 **"최종 승인(통과)"**을 하면:
   - 해당 브랜치의 작업 내역을 커밋(`git add . && git commit -m "feat(T-XXX): ..."`)합니다.
   - `main` 브랜치로 이동하여 병합(`git checkout main && git merge feature/...`)합니다.
   - `docs/TASKS.md`의 작업 상태를 `[done]`으로 변경합니다.
   - 다음 `[todo]` 태스크를 제안하고 다시 Step 1으로 돌아갑니다.

## 3. 핵심 규칙 요약

- 💡 한 번에 여러 개의 태스크를 뭉뚱그려 개발하지 마세요. **반드시 1개의 태스크 단위로 끊어서** 테스트를 받으세요.
- 💡 사용자의 명시적인 "**승인(OK)**" 없이 멋대로 다음 `[todo]` 태스크로 넘어가지 마세요.
